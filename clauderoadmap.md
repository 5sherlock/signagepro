# SignagePro 프로젝트 로드맵
> 오설록 제주점 디지털 사이니지 원격 관리 시스템
> 작성일: 2026-05-10 | 작성: Claude Sonnet 4.6 (Anthropic)

## 1. 프로젝트 개요

### 배경
- **사업장**: 제주 오설록 매장
- **디지털 보드**: 5개 (각 보드에 크라이저 셋톱박스 HDMI 연결)
- **네트워크**: 유선 LAN (고정 IP 5개) 또는 와이파이
- **목적**: 이미지/동영상 콘텐츠를 원격으로 5개 보드에 동시 송출

### 기존 업체 변경 이력
| 시기 | 업체 | 비고 |
|---|---|---|
| 이전 | 튠미디어 (tunemedia.co.kr) | 크라이저 셋톱박스 사용 |
| 현재 | 와이브로드 (ybroad.net) | Y-Board Premium Manager 사용 |
| **목표** | **자체 개발 (SignagePro)** | **antigravity Gemini 연동** |

### 셋톱박스 사양
- **제조사**: (주)크라이저 (krizer.com)
- **OS**: Android 8.1
- **CPU**: Quad-Core ARM
- **연결**: HDMI → 디지털 보드 / 유선LAN or Wi-Fi → 서버
- **현재 앱**: 와이브로드 Y-Board Premium 클라이언트

## 2. 현재 시스템 문제점

### 핵심 문제 3가지

#### ① 와이파이 접속 불안정
- 와이파이 환경에서 셋톱박스 접속이 자주 끊김
- 임시방편으로 유선 고정 IP 5개를 확보하여 운영 중
- 근본적인 재접속 로직 및 오프라인 캐시 대응이 없음

#### ② 멀티 디바이스 싱크 불일치
- 5개 보드가 동시에 같은 콘텐츠를 재생해야 하나 타이밍이 맞지 않음
- Y-Board는 단순 IP 기반 푸시 방식으로 실시간 동기화 프로토콜 부재
- 고객 입장에서 보드마다 다른 화면이 표시되는 문제

#### ③ 콘텐츠 롤링(순환 재생) 불가
- 여러 이미지/동영상을 순서대로 순환 재생하는 스케줄 기능 제한
- 콘텐츠 교체 시 수동 작업 필요

### 와이브로드 Y-Board Premium 한계
- 온프레미스(로컬 PC) 설치형 관리 프로그램 (`C:\Program Files (x86)\Y-Board-Premium-Manager`)
- 멀티 디바이스 동기화 기능 미흡
- 커스터마이징 불가 (타사 솔루션)
- 지속적인 라이선스 비용 발생

### 핵심 트렌드
1. **클라우드 기반 CMS가 표준**: 중앙 집중식 콘텐츠 관리, 실시간 배포

5. **시장 성장률**: CAGR 7.8% (2025~2032)

- 2~3년 전 기술 수준의 온프레미스 방식
- 자체 개발(SignagePro)은 시황 트렌드에 완전히 부합

## 4. SignagePro 시스템 아키텍처

### 전체 구조
```
[사업장/본사]                    [인터넷]              [오설록 매장]
관리 Web UI  ──→  서버 API  ──→  WebSocket  ──→  셋톱박스 Android App (×5)
(콘텐츠 업로드)   (Node.js)     (실시간 동기화)       ↓
                    ↓                              로컬 캐시 (오프라인 대응)
                  DB/Storage                           ↓
               (파일 저장소)                      HDMI → 디지털 보드
```

### 온프레미스 vs 클라우드 선택
| 구분 | 온프레미스 | 클라우드 |
|---|---|---|
| **설명** | 사무실/매장 PC에 서버 직접 설치 | AWS, GCP 등 외부 서버 사용 |
| **비용** | 초기 셋업 후 무료 | 월 사용료 발생 |
| **관리** | 직접 유지보수 | 공급자가 관리 |
| **확장성** | 제한적 | 유연함 |
| **추천** | 초기 개발/테스트 단계 | 운영 안정화 후 |

> **초기 추천**: 온프레미스(로컬 PC 서버)로 개발 및 테스트 → 안정화 후 클라우드 이전

## 5. 개발 목표 및 기술 스택

### 서버 (Backend)
- **언어**: Node.js (Express) 또는 Python (FastAPI)
- **실시간 통신**: WebSocket (Socket.io)
- **스케줄러**: cron 기반 재생 큐
- **DB**: PostgreSQL 또는 SQLite (소규모)
- **파일 저장**: 로컬 스토리지 또는 S3 호환

### 클라이언트 (셋톱박스 Android App)
- **개발 방식**: React Native 또는 Flutter (크로스플랫폼)
  - 또는 Android Native (Java/Kotlin) - 셋톱박스 최적화
- **핵심 기능**:
  - WebSocket 연결 및 명령 수신
  - 이미지/동영상 로컬 캐시
  - NTP 기반 시간 동기화 (싱크)
  - 재접속 자동화 (네트워크 불안정 대응)

### 관리 대시보드 (Frontend)
- **프레임워크**: React.js
- **기능**: 콘텐츠 업로드, 디바이스 상태 모니터링, 스케줄 관리

### AI 연동
- **플랫폼**: antigravity Gemini
- **활용**: 콘텐츠 추천, 자동 스케줄 최적화, 이상 감지
